ADR-001: Growlog AI — журнал выращивания, контекстная база знаний и движок Grow Report

Статус: Proposed
Дата: 2026-03-22
Автор: A. Safiulin, OG Lab.

⸻

Контекст

Growlog AI создаётся как PWA-приложение для гроверов и руководителей выращивания. Продукт должен решать сразу несколько задач:
	1.	Дать гроверу удобный способ быстро фиксировать происходящее на ферме:
	•	голосом,
	•	текстом,
	•	фотографиями,
	•	через автоматический приём данных от датчиков и внешних API.
	2.	Собирать непрерывную историю цикла выращивания:
	•	события,
	•	параметры среды,
	•	наблюдения,
	•	фото,
	•	действия,
	•	отклонения,
	•	комментарии,
	•	выполнение SOP,
	•	а также данные за предыдущие циклы.
	3.	Использовать всю накопленную историю как контекстную модель мира фермы для встроенного AI-ассистента, чтобы он:
	•	отвечал на вопросы,
	•	объяснял отклонения,
	•	давал рекомендации,
	•	учитывал весь цикл и, по возможности, прошлые циклы,
	•	не опирался только на последний диалог или отдельную запись.
	4.	Формировать Grow Report:
	•	как красивый PDF с комментариями и коллажами фото,
	•	как HTML-страницу с продуманной подачей материалов,
	•	как отчёт по текущему состоянию для руководителя,
	•	как удобную хронологию событий по дням.
	5.	Поддерживать SOP management и контроль исполнения:
	•	хранить SOP,
	•	напоминать о регламентных действиях по дате, событию или ситуации,
	•	фиксировать намерение исполнителя,
	•	собирать данные о фактическом исполнении,
	•	использовать эту информацию в общей картине происходящего на ферме.
	6.	Поддерживать голосовой интерфейс не только на ввод, но и на вывод, чтобы продукт был ближе к диалогу, а не только к форме журнала.

Дополнительный важный контекст: профессиональные гроверы часто не доверяют AI-рекомендациям из-за галлюцинаций, поверхностности и отсутствия реального контекста. Поэтому продукт должен быть спроектирован так, чтобы:
	•	советы были основаны на фактах из журнала,
	•	использовались профессиональные источники знаний,
	•	система умела явно показывать степень уверенности,
	•	при нехватке данных она запрашивала уточнение, а не выдумывала.

⸻

Проблема

Если построить Growlog AI как обычный чат с AI и набором разрозненных заметок, то система не сможет:
	•	reliably учитывать весь цикл выращивания,
	•	понимать причинно-следственные связи,
	•	сопоставлять события с сенсорными данными,
	•	отслеживать соблюдение SOP,
	•	создавать качественные отчёты,
	•	заслужить доверие профессиональных пользователей.

Нужна архитектура, в которой журнал, события, датчики, фото, SOP и AI-слой работают как единая система.

⸻

Решение

Мы принимаем решение строить Growlog AI как event-centric grow operating journal с двумя основными продуктовыми целями:

1. Контекстный AI-помощник по выращиванию

Система должна непрерывно собирать и структурировать историю фермы, чтобы встроенный агент мог использовать:
	•	данные текущего цикла,
	•	историю предыдущих циклов,
	•	наблюдения пользователя,
	•	данные датчиков,
	•	фото,
	•	протоколы SOP,
	•	справочную профессиональную базу знаний.

Рекомендации AI должны генерироваться не из “общих знаний модели”, а из комбинации:
	•	фактов из локальной истории фермы,
	•	текущего контекста,
	•	проверенных профессиональных источников,
	•	явной логики извлечения релевантного контекста.

2. Система формирования Grow Report

Система должна уметь из той же базы данных формировать:
	•	дневной статус-отчёт,
	•	отчёт по циклу,
	•	отчёт для руководителя,
	•	публичный или полу-публичный grow report,
	•	PDF-отчёт с фото-коллажами,
	•	HTML-отчёт с аккуратной визуальной структурой и скрываемыми фотоблоками.

⸻

Принципы решения

1. Источник правды — единый журнал событий

Все факты должны храниться как события или связанные сущности:
	•	пользовательская запись,
	•	сенсорное измерение,
	•	фото,
	•	комментарий,
	•	действие,
	•	задача,
	•	выполнение SOP,
	•	AI-анализ,
	•	системное напоминание.

2. AI должен работать поверх фактов, а не вместо фактов

LLM не является источником истины. Источник истины — это:
	•	журнал событий,
	•	структурированные параметры,
	•	фотоархив,
	•	SOP,
	•	верифицированная база знаний.

3. Фото — полноценный тип данных

Фотографии используются для анализа, истории и отчётов.

4. SOP — часть операционной модели фермы

SOP интегрированы в ядро системы и влияют на аналитику.

5. Продукт = журнал + ассистент + управление

6. Сначала детерминированные механики, потом сложная агентность

Для MVP и первых production-версий мы предпочитаем:
	•	явные типы событий,
	•	предсказуемые пайплайны обработки,
	•	правила и шаблоны поверх "магии",
	•	подтверждение пользователем важных интерпретаций.

Слабая модель или джуниор-разработчик должны быть способны реализовать систему по ADR без необходимости "додумывать архитектуру".

⸻

Архитектурное решение

Система разделяется на 5 слоёв:

1. Capture Layer
	•	ввод голосом,
	•	ввод текстом,
	•	загрузка фото,
	•	ручной ввод параметров,
	•	импорт сенсорных данных,
	•	фиксирование исполнения SOP.

2. Operational Data Layer
	•	единый журнал событий,
	•	структурированные сущности цикла,
	•	медиа-архив,
	•	SOP и их исполнения,
	•	очередь фоновой обработки.

3. Intelligence Layer
	•	транскрибация,
	•	извлечение структурированных фактов,
	•	анализ изображений,
	•	поиск аномалий,
	•	сбор контекста для ответа,
	•	генерация объяснений и рекомендаций.

4. Presentation Layer
	•	дневной обзор,
	•	таймлайн,
	•	страница SOP Management,
	•	страница ассистента,
	•	конструктор и экспорт Grow Report.

5. Trust Layer
	•	grounding ответа,
	•	показывание уверенности,
	•	явный список недостающих данных,
	•	запрет на скрытые догадки в критичных рекомендациях.

Технологический стек:

Клиент
	•	Next.js
	•	TypeScript
	•	PWA
	•	shadcn/ui
	•	TanStack Query
	•	Zustand только для локального UI-state, а не как источник бизнес-данных

Backend
	•	Supabase (Postgres + Storage + Auth)
	•	Supabase Edge Functions для AI-интеграций и webhooks
	•	Postgres cron / scheduled jobs для напоминаний и фоновых пересчётов

AI
	•	Whisper или совместимый STT для голоса
	•	LLM для диалога, структурирования заметок и генерации отчётов
	•	Vision model для анализа фото
	•	embeddings + retrieval для базы знаний и исторического контекста

Deployment
	•	Vercel для web-клиента
	•	Supabase для БД, storage и auth

Архитектурное правило:

LLM не пишет напрямую в критичные таблицы по собственному усмотрению. Любое изменение данных происходит только через явные backend use cases:
	•	create_log_entry
	•	create_sensor_reading
	•	create_sop_execution
	•	create_ai_insight
	•	generate_report

Это защищает систему от "агент сам что-то додумал и сохранил как факт".

Модель данных высокого уровня

Обязательные сущности первого уровня:

1. Farm
	•	логическая ферма или домашний сетап
	•	владелец, таймзона, геолокация, единицы измерения

2. Grow Cycle
	•	отдельный цикл выращивания
	•	сорт, дата старта, текущая стадия, статус, ожидаемые даты этапов

3. Scope
	•	область, к которой относится запись
	•	типы: farm, room, tent, zone, bed, reservoir, plant_group, plant
	•	нужна, чтобы одни и те же механики работали и для домашнего бокса, и для фермы

4. Event Log
	•	центральная таблица фактов
	•	каждая запись имеет:
	•	event_type,
	•	source_type,
	•	occurred_at,
	•	cycle_id,
	•	scope_id,
	•	author_type,
	•	raw_text,
	•	structured_payload,
	•	linked_assets_count

5. Sensor Reading
	•	измерение с числовыми значениями
	•	например: temp_air, rh, ph, ec, ppm, vpd, runoff_ph, runoff_ec
	•	может приходить как отдельное событие и как структурированная таблица показаний

6. Photo Asset
	•	фото или серия фото
	•	storage path,
	•	taken_at,
	•	scope_id,
	•	cycle_id,
	•	capture_source,
	•	vision_summary,
	•	derived_tags

7. SOP Definition
	•	шаблон регламента
	•	название,
	•	цель,
	•	инструкция,
	•	trigger_type,
	•	trigger_rules,
	•	required_inputs_after_execution,
	•	severity_if_missed,
	•	active_flag

8. SOP Run
	•	конкретное ожидаемое исполнение регламента
	•	создаётся автоматически из SOP Definition и контекста цикла
	•	имеет due_at / due_window / scope / status

9. SOP Execution
	•	факт выполнения или невыполнения конкретного SOP Run
	•	status: done, skipped, delayed, blocked, partially_done
	•	notes, evidence_photos, measured_values, performer_feedback

10. AI Insight
	•	машинный вывод, который не считается первичным фактом
	•	типы: summary, anomaly, recommendation, causal_hypothesis, report_block
	•	обязан ссылаться на исходные данные

11. Knowledge Document
	•	проверенный внешний документ или внутренняя статья
	•	используется для retrieval и обоснований рекомендаций

12. Report
	•	сгенерированный артефакт отчёта
	•	типы: daily, cycle, manager, public_html, pdf
	•	хранит snapshot данных на момент генерации

Минимальные связи:
	•	Farm 1->N Grow Cycle
	•	Grow Cycle 1->N Scope
	•	Grow Cycle 1->N Event Log
	•	Event Log N->N Photo Asset
	•	SOP Definition 1->N SOP Run
	•	SOP Run 1->N SOP Execution
	•	AI Insight N->N Event Log
	•	Report N->N Event Log

Правило моделирования:

Если что-то произошло во времени, это должно иметь представление в `Event Log`, даже если есть отдельная специализированная таблица. Специализированные таблицы нужны для удобства запросов и аналитики, но не заменяют журнал.

Основные пользовательские сценарии

1. Голосовой лог
	•	пользователь нажимает запись
	•	говорит: что произошло, что сделал, что заметил
	•	система расшифровывает речь
	•	LLM извлекает структуру события
	•	пользователь подтверждает или быстро правит результат
	•	запись сохраняется в журнал

2. Фото-лог
	•	пользователь загружает фото одного растения, группы или общего вида
	•	задаёт scope и при необходимости комментарий
	•	vision pipeline создаёт теги и краткое описание
	•	система связывает фото с днём цикла, scope и соседними событиями

3. SOP execution dialog
	•	система напоминает о регламенте
	•	пользователь отмечает done / delayed / skipped / blocked
	•	если нужно, система задаёт обязательные follow-up вопросы
	•	исполнение попадает в журнал и влияет на аналитику

4. Ask the farm
	•	пользователь задаёт вопрос в свободной форме
	•	система собирает факты по текущему циклу, scope, SOP и истории
	•	ответ формируется с опорой на конкретные факты и знания
	•	если данных мало, система сначала спрашивает уточнение

5. Generate Grow Report
	•	пользователь выбирает цикл и тип отчёта
	•	система собирает таймлайн, фото, ключевые события и выводы
	•	генерирует HTML-страницу и/или PDF
	•	HTML-должен быть пригоден для ручной публикации и копипаста на площадки вроде [Dzagi](https://dzagi.club)

SOP Management

SOP Management является самостоятельным модулем, а не просто чеклистом.

SOP Definition обязана поддерживать следующие типы триггеров:
	•	date_based: конкретная дата или набор дат
	•	time_based: время суток или окно времени
	•	stage_based: стадия цикла, например early_veg или week_3_flower
	•	offset_based: через N дней после старта, пересадки, перевода на 12/12, промывки и т.д.
	•	event_based: после события определённого типа
	•	recurring_daily: ежедневно
	•	recurring_interval: каждые N часов / дней
	•	condition_based: если параметр вышел за порог
	•	location_based: когда пользователь находится на месте

Для `location_based` на MVP допускается упрощение:
	•	не строим сложный background geofencing engine,
	•	показываем location-sensitive SOP только при открытии приложения на ферме или при явном check-in.

Каждый SOP обязан содержать:
	•	что нужно сделать
	•	зачем это нужно
	•	какие данные надо зафиксировать после выполнения
	•	какой допустимый дедлайн
	•	что считать провалом исполнения

Система не только напоминает, но и ведёт диалог исполнения:
	•	"Сделал ли ты промывку?"
	•	"Каким был runoff EC?"
	•	"Есть ли фото после процедуры?"
	•	"Были ли отклонения?"

Эти ответы сохраняются как факты и используются в последующих рекомендациях.

⸻

Контекст для AI-ответа

Каждый AI-ответ строится из 6 блоков контекста:

1. Текущий вопрос пользователя
2. Краткое состояние текущего цикла
3. Последние релевантные события по нужному scope
4. История похожих ситуаций из прошлых циклов
5. Связанные SOP и статус их исполнения
6. Релевантные knowledge documents

Пайплайн ответа:
	•	retrieve facts
	•	retrieve SOP context
	•	retrieve historical analogs
	•	retrieve professional knowledge
	•	build answer context
	•	generate answer
	•	store answer as AI Insight only if он полезен как артефакт истории

Обязательное правило:

Если запрос содержит диагностику проблемы, система сначала должна ответить:
	•	что она наблюдает по фактам,
	•	чего ей не хватает,
	•	какие 2-4 гипотезы вероятны,
	•	какое следующее наблюдение или измерение снизит неопределённость.

🧠 ADVANCED INTELLIGENCE LAYER

⸻

🧠 WHY ENGINE — объяснение причин

Система обязана объяснять причинно-следственные связи.

WHY ENGINE должен работать на уровне "объясни почему", а не только "посоветуй что делать".

Он обязан уметь связывать:
	•	события,
	•	сенсорные данные,
	•	исполнение или пропуск SOP,
	•	визуальные сигналы на фото,
	•	исторические паттерны цикла.

Формат результата WHY ENGINE:
	•	observation: что произошло
	•	evidence: на каких фактах основан вывод
	•	possible_causes: список гипотез
	•	most_likely_cause: наиболее вероятная причина
	•	what_to_check_next: как подтвердить или опровергнуть гипотезу

На MVP причинность строится эвристически:
	•	по временной близости,
	•	по пороговым правилам,
	•	по совпадению известных шаблонов,
	•	по retrieval из похожих случаев.

⸻

🧠 GROW MEMORY

Система формирует долгосрочную память:

GROW MEMORY должна хранить не просто события, а устойчивые выводы о ферме:
	•	повторяющиеся паттерны,
	•	типичные привычки оператора,
	•	частые ошибки,
	•	рабочие стратегии,
	•	реакции конкретных сортов и сетапов.

Память хранится на трёх уровнях:
	•	farm memory
	•	cycle memory
	•	cultivar memory

Примеры memory items:
	•	"в этом боксе ночью стабильно падает температура ниже целевого диапазона"
	•	"на 3 неделе цветения пользователь часто недополивает"
	•	"сорт X в этом сетапе лучше переносит lower EC"

Memory item не является истиной навсегда. Он должен иметь:
	•	supporting evidence count
	•	last_confirmed_at
	•	confidence
	•	status: active / stale / rejected

⸻

📸 PHOTO TIMELINE INTELLIGENCE

Система обязана сравнивать изображения во времени и по scope.

Минимальные функции:
	•	группировка фото по дню цикла и scope
	•	сравнение "сегодня против 3 дня назад"
	•	сравнение "начало стадии против текущего состояния"
	•	выделение фото-кандидатов для отчёта

Выходы photo intelligence:
	•	визуальные теги
	•	краткое описание изменений
	•	подборка лучших фото для Grow Report
	•	сигналы "нужна проверка человеком"

⸻

📊 ANOMALY DETECTION

Система должна детектировать:
	•	отклонения показателей от целевого диапазона
	•	несовпадение действий и ожидаемой реакции
	•	пропуски важных SOP
	•	аномальную динамику по фото и событиям

MVP-правило:

Anomaly Detection сначала rule-based, а не "умная магия". Источники аномалий:
	•	threshold breach
	•	missing expected action
	•	mismatch between stage and measurements
	•	repeated negative pattern from grow memory

Каждая аномалия должна иметь:
	•	severity
	•	detected_at
	•	triggering facts
	•	recommended next action

⸻

🧾 AUTO-STORY MODE

AUTO-STORY MODE генерирует narrative reports из фактов таймлайна.

Он не должен выдумывать "красивую историю". Его задача:
	•	собрать важные моменты цикла,
	•	описать развитие по стадиям,
	•	подтянуть ключевые фото,
	•	вынести решения, ошибки и результаты.

Форматы:
	•	short daily recap
	•	cycle chapter summary
	•	public grow report text
	•	manager summary

Для публикационных отчётов HTML должен:
	•	быть читаемым без приложения,
	•	сворачивать длинные фото-блоки,
	•	группировать фото по дню или по этапу,
	•	помогать пользователю вручную переносить материал на внешние сайты.

⸻

🎤 VOICE ASSISTANT

Система поддерживает и voice input, и voice output.

Voice input используется для:
	•	создания записей,
	•	быстрого ответа на SOP check-in,
	•	вопросов ассистенту на ходу.

Voice output используется для:
	•	озвучивания ежедневного фокуса,
	•	зачитывания ответа ассистента,
	•	пошагового диалога по SOP.

MVP-ограничение:

Голосовой режим не должен быть отдельной параллельной архитектурой. Он является альтернативным интерфейсом поверх тех же use cases, что и текстовый режим.

⸻

🛡️ TRUST LAYER

Все AI-ответы обязаны содержать:
	•	grounding: на какие факты и документы опирается ответ
	•	confidence: высокая / средняя / низкая
	•	missing_data: каких данных не хватает

Запрещено:
	•	выдавать догадку за измеренный факт
	•	утверждать диагноз без опоры на наблюдение
	•	молчать о низкой уверенности

Шаблон безопасного ответа:
	•	что известно
	•	что вероятно
	•	что неизвестно
	•	что сделать следующим шагом

⸻

🧭 DAILY FOCUS

Система показывает:
	•	что важно сегодня
	•	какие SOP выполнить
	•	риски
	•	рекомендации

`Daily Focus` является главной операционной страницей.

Она должна собирать:
	•	SOP due today
	•	overdue SOP
	•	новые аномалии
	•	важные изменения с прошлого визита
	•	1-3 главные рекомендации
	•	быстрые действия: записать лог, загрузить фото, отметить SOP

Приоритет сортировки:
	1. критичные overdue задачи
	2. сегодняшние обязательные SOP
	3. аномалии высокой важности
	4. рекомендованные, но не обязательные действия

⸻

Этапы реализации

1. Foundation MVP
	•	auth
	•	farms / cycles / scopes
	•	event log
	•	photo upload
	•	manual sensor entry
	•	daily focus without AI

Результат этапа: можно полноценно вести журнал цикла без нейросети.

2. Voice + Structured Logging
	•	voice capture
	•	transcription
	•	structured event extraction
	•	user confirmation before save

Результат этапа: голос становится быстрым способом ввода фактов.

3. SOP Management
	•	SOP definitions
	•	trigger engine
	•	SOP runs
	•	execution dialog
	•	daily reminders

Результат этапа: система реально сопровождает операционную дисциплину.

4. AI Advisor
	•	context assembly
	•	question answering over facts + SOP + knowledge
	•	why engine
	•	trust layer

Результат этапа: ассистент отвечает на вопросы по текущей ферме, а не "вообще".

5. Vision + Photo Intelligence
	•	photo tagging
	•	image comparison over time
	•	photo-based insights
	•	report photo selection

Результат этапа: фото становятся аналитическим сигналом, а не просто архивом.

6. Reports Engine
	•	daily report
	•	cycle report
	•	public HTML grow report
	•	PDF export

Результат этапа: из журнала автоматически получаются артефакты для управления и публикации.

Порядок этапов выбран намеренно:
	•	без Foundation нет фактов,
	•	без фактов AI бесполезен,
	•	без SOP продукт не становится операционной системой выращивания,
	•	без photo intelligence и reports не раскрывается контентная ценность продукта.

⸻

Не-цели первого релиза

В первый релиз не входят:
	•	полная автоматизация IoT-интеграций со всеми возможными девайсами
	•	автономный агент, который сам принимает действия без подтверждения человека
	•	сложная предиктивная биологическая модель урожайности
	•	публичная социальная сеть внутри продукта
	•	полностью автоматическая публикация на внешние форумы от имени пользователя

Это важно, чтобы не размыть основу: журнал, SOP, контекстный ассистент и отчёты.

⸻

Последствия

Плюсы
	•	единая архитектура для журнала, SOP, AI и отчётов
	•	высокая ценность для ежедневной работы гровера
	•	рост доверия к советам за счёт grounding и памяти фермы
	•	простая декомпозиция на таблицы, use cases и фоновые задачи
	•	контентная ценность через красивые grow reports

Минусы
	•	высокая сложность доменной модели
	•	нужно аккуратно проектировать event taxonomy
	•	без хорошего UX подтверждения пользователи могут уставать от структурирования
	•	vision и retrieval увеличивают стоимость AI-инфраструктуры

Риски
	•	если journal layer будет плохим, вся AI-часть станет шумной
	•	если SOP будут слишком сложными в настройке, их не будут заполнять
	•	если trust layer будет формальным, а не реальным, профессионалы быстро потеряют доверие

⸻

Итог

Growlog AI = операционная память фермы + контекстный AI-агроном + SOP-координатор + движок Grow Report.

Ключевая идея ADR:

Мы строим не "чат про выращивание" и не "дневник заметок", а систему, где каждая запись, фотография, замер, регламент и диалог формируют обновляемую модель фермы. Именно на этой модели должны строиться советы, напоминания, объяснения и отчёты.
