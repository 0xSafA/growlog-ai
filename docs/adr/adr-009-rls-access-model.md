ADR-009: Growlog AI — RLS и модель доступа (безопасность и SaaS-архитектура)

Статус: Proposed
Дата: 2026-03-22
Автор: A. Safiulin, OG Lab.
Связан с: ADR-001, ADR-002, ADR-004, ADR-007, `docs/implementation/data-platform-implementation-spec.md`

⸻

Контекст

Growlog AI разворачивается как мультитенантный SaaS на Supabase (Postgres + Auth + Storage). Каноническая граница изоляции данных — **ферма** (`farms`): операционный журнал, сенсоры, медиа, SOP, отчёты и производные слои привязаны к `farm_id` (см. ADR-002).

Клиентское приложение по умолчанию использует **anon key** и сессию пользователя (`auth.uid()`). Серверные процессы (Edge Functions, фоновые задачи, пайплайны AI) могут использовать **service role**, который в Supabase **обходит RLS**. Это создаёт классический разрыв: политика безопасности должна быть явной на трёх уровнях — **БД (RLS)**, **backend use cases** и **продуктовые правила** (например, trust layer из ADR-004).

⸻

Проблема

1. **Утечки между тенантами** — любая ошибка в запросе или в коде клиента без жёсткой привязки к `farm_id` приводит к cross-tenant data exposure.
2. **Слепое доверие service role** — обход RLS ускоряет разработку, но переносит всю проверку в приложение; одна забытая проверка даёт полный доступ ко всем фермам.
3. **Роли и зоны ответственности** — в `farm_users.role` заложены `grower`, `manager`, `admin`, `viewer`, но политика «кто что видит и меняет» должна быть согласована между RLS, API и UX (ADR-005).
4. **Публичные артефакты** — отчёты и HTML для внешней аудитории нельзя решать ослаблением RLS на «внутренних» таблицах; нужен отдельный publish layer (ADR-007, implementation spec).
5. **Storage** — файлы в bucket’ах должны быть изолированы так же строго, как строки в таблицах.

⸻

Решение

Мы принимаем **модель доступа с приоритетом Row Level Security в Postgres** и явным разделением ролей клиента и сервиса.

### 1. Принципы

1. **RLS включён** для всех рабочих таблиц с пользовательскими данными; отсутствие политики трактуется как «нет доступа», а не «открыто всем».
2. **Тенант по умолчанию — ферма**: видимость и изменяемость строк определяются членством в `farm_users` для соответствующего `farm_id`.
3. **Клиент не является доверенным**: прямой доступ к БД только с ключом, который подчиняется RLS (anon + JWT пользователя); критичные массовые операции и bootstrap — через **security definer** RPC или доверенный backend с повторной проверкой домена.
4. **Service role не отменяет доменную модель**: код с service role обязан **повторять** те же инварианты, что и RLS (проверка `farm_id`, роли, scope), а не полагаться на «мы уже на сервере».
5. **Публичное ≠ открыть таблицу**: публичные отчёты и снапшоты — через отдельные сущности/артефакты, подписанные URL, или материализованный publish слой с собственной политикой (см. п. 6).

### 2. Механика членства (MVP foundation)

Канонический способ выразить «пользователь имеет доступ к ферме»:

- Таблица `farm_users` связывает `auth.users.id` и `farm_id` с полем `role`.
- Вспомогательная функция **`user_farm_ids()`** (security definer, `stable`, `search_path = public`) возвращает множество `farm_id`, где состоит текущий пользователь:

```sql
select farm_id from public.farm_users where user_id = auth.uid();
```

Политики RLS на таблицах с `farm_id` используют паттерн:

- `using` / `with check`: `farm_id in (select public.user_farm_ids())`

Исключения документируются отдельно (например, справочные `sensor_metrics` с `farm_id is null` как глобальные метрики — чтение разрешено всем аутентифицированным членам любой фермы при необходимости; запись — только в контексте своей фермы).

### 3. Создание фермы и стартовая запись

Прямой `insert` в `farms` и `farm_users` с клиента **запрещён** политиками (`with check (false)` на insert), чтобы не создавать «осиротевшие» фермы и произвольное назначение ролей.

Создание фермы и первой строки членства выполняется через **RPC** `create_farm_with_membership` (security definer): при валидном `auth.uid()` создаётся ферма и запись `farm_users` с ролью `admin`.

### 4. Матрица ролей и эволюция политик

В модели данных зафиксированы роли: `grower`, `manager`, `admin`, `viewer`.

- **Foundation MVP** в миграциях может применять **единую политику по членству** (все роли с записью в `farm_users` имеют одинаковый доступ к строкам своей фермы на уровне RLS). Это упрощает первый релиз и снижает риск ошибок в политиках.
- **Целевое состояние продукта**: дифференциация по `role` и опционально по scope (например, менеджер — шире read по циклу/зоне; viewer — только чтение; grower — ограниченный write). Реализация — отдельные `using`/`with check` с подзапросом к `farm_users.role` и при необходимости таблицам назначений (SOP assignments, zone ownership).

**Правило проектирования:** любое новое правило «X не может удалять Y» сначала выражается в доменных use cases и тестах, затем дублируется или усиливается в RLS, чтобы клиент с подставным запросом не обошёл продуктовую логику.

### 5. Storage (фото и медиа)

Изоляция на уровне объектов Storage согласована с фермой:

- Bucket не публичный (`public = false`).
- Путь объекта включает идентификатор фермы (например, первый сегмент пути = `farm_id::text`).
- Политики `storage.objects` проверяют, что этот сегмент принадлежит ферме из `user_farm_ids()` для текущего пользователя.

Так медиа не просачивается между тенантами даже при угадывании UUID объектов вне приложения.

### 6. Публичные отчёты и внешние читатели

Внутренние таблицы `reports`, `report_artifacts`, сырые `events` **не** открываются для анонимного доступа через ослабление RLS.

Допустимые паттерны:

- Отдельная таблица/представление **published snapshots** с собственными политиками (например, доступ по `share_token` или по роли `anon` только к зашифрованному/ограниченному JSON).
- Генерация **подписанного URL** на объект в отдельном bucket/prefix для публичной версии.
- Отдача HTML/PDF **через** Edge Function или API route, где авторизация проверяется явно, а данные читаются service role после валидации токена публикации.

Это согласуется с ADR-007 и разделом implementation spec про publish layer.

### 7. AI, retrieval и service role

- **Retrieval и сбор контекста** (ADR-003) должны получать только те `farm_id` / `scope_id`, к которым пользователь имеет право; при вызове из клиента это естественно следует из RLS при чтении из БД.
- **Пайплайны с service role** (индексация, enrichment, пересчёт derived) выполняются в контексте **конкретной фермы**, переданной из задания (job payload), с проверкой, что задание создано доверенным путём.

ADR-004 фиксирует trust layer на уровне вывода AI; **ADR-009** фиксирует, что доступ к сырью для контекста не должен расширяться за счёт обхода RLS без явного аудита.

### 8. Аудит и соответствие (направление)

Для зрелого SaaS ожидаются:

- Логирование административных действий (`farm_users` изменения, экспорт данных).
- Опционально: отдельная таблица audit events с записью через security definer триггеры или централизованный backend.

Детали выносятся в отдельные ADR или operational runbooks, но **граница тенанта** остаётся неизменной.

⸻

Последствия

**Положительные**

- Явная модель «одна ферма — один остров данных» на уровне СУБД.
- Клиент на anon key не может перечислить чужие фермы при корректных политиках.
- Storage согласован с табличными данными.
- Путь эволюции ролей (от flat membership к RBAC/ABAC) заложен в схему `farm_users`.

**Отрицательные / стоимость**

- Сложнее отладка запросов (нужно помнить про `auth.uid()` и политики).
- Дублирование проверок между RLS и server-side кодом — осознанная цена; избегать только RLS или только application checks нельзя.
- Политики по ролям усложняют миграции; изменения должны сопровождаться тестами на изоляцию.

⸻

Что не фиксируется этим ADR

- Конкретный DDL каждой будущей таблицы (см. ADR-002 и миграции).
- Формат share links и TTL подписанных URL для публичных отчётов.
- Организационный уровень выше фермы (org / billing account) — при появлении потребуется ADR о иерархии тенантов и наследовании доступа.
- Детальная матрица IAM для enterprise (SSO, SCIM) — вне scope текущего документа.

⸻

Короткая формулировка решения

**Изоляция данных между фермами обеспечивается Row Level Security в Postgres и согласованными политиками Storage; пользовательский доступ привязан к `farm_users`; критичные операции создания тенанта и доверенные записи идут через security definer RPC или проверенный backend; service role не заменяет доменные проверки; публичный контент выдаётся через publish layer, а не ослаблением RLS на операционных таблицах.**
